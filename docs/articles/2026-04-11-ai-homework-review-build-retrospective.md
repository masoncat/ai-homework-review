# 从“假演示”到“真实可用”：一个 AI 作业批改网站的完整开发复盘

做 AI 产品最容易踩的一个坑，不是模型效果不够好，而是“看起来像通了，其实没通”。页面能点，按钮会动，流程会跳转，甚至还能返回一段像模像样的讲评文案，但只要上传没有真正进对象存储、OCR 没有真正识别图片、文本模型没有真正生成讲评，这个产品本质上还是一个“演示壳”。

这次我们做的，就是把一个 AI 作业批改站，从“前端能演示”一路推进到“真实链路完全打通”。最终结果不是模拟成功，而是老师在首页输入体验码、上传真实答题卡图片、后端调用真实 OCR 模型识别答案、再调用真实文本模型生成讲评，最后返回真实分数、错题和讲评建议。整个闭环跑通以后，这个项目才真正具备了继续迭代的基础。

[配图1：项目首页最终效果图]

一开始最明显的问题，是“体验入口能点，但并没有走真实后端”。用户点击“立即体验批改演示”时，页面看起来进入了流程，但实际不是走真实 AI 接口，而是依赖前端演示数据或者伪结果。这种实现方式适合早期验证页面结构，却不适合继续做产品，因为一旦前端、后端、对象存储、模型服务分离之后，每一段链路都可能单独失败。只有把每一段都变成真实调用，才能知道系统到底卡在哪里。

因此第一步不是继续优化页面，而是把“假接口”全部换成“真接口”。我们明确了后端的职责：会话签发、上传策略下发、对象存储读写、OCR 调用、判分、讲评生成，全部由服务端协调。前端只做四件事：输入体验码、填写标准答案、上传答题卡、展示结果。这样职责边界清晰，后续排障也更容易。

[配图2：前后端架构草图]

模型选型也是这一轮开发里比较关键的决策。作业批改并不是一个单模型全包的场景。OCR 识别和讲评生成是两个完全不同的问题。前者重点是看图、识别固定版式题号与答案，后者重点是根据判分结果输出自然语言讲评。因此我们最后采用了双模型方案：OCR 使用阿里云 DashScope 的 `qwen-vl-ocr-latest`，讲评生成使用 `gpt-5.4`。这样的拆分有两个好处。第一，模型各司其职，识别和表达都能用各自更适合的能力。第二，后续如果要替换某一侧的模型，不需要影响整条链路。

当真实 AI 接口接上之后，第二个难点不是模型本身，而是“图片到底怎么到模型手里”。前端上传答题卡，后端 OCR 要读取这张图，如果中间用的是本地模拟文件或者内存数据，那只是开发捷径，不是真实生产方案。于是我们把上传链路切到了阿里云 OSS，并进一步采用 STS 临时凭证的方式，让前端尽量走直传。

这里的核心设计是两个身份分离。一个是本地开发用的 RAM 用户，它代表后端服务本身，负责读取图片、调用 STS、生成签名 URL。另一个是 `homework` 角色，它不存长期密钥，而是被服务端临时扮演，用来给前端下发上传临时凭证。这种分离非常重要，因为前端上传不应该直接暴露后端长期 AK/SK，而后端读取图片又确实需要一个稳定身份来完成。很多人第一次配阿里云权限时会把 RAM 用户和 RAM 角色混在一起，结果不是 `AssumeRole` 报错，就是 `GetObject` 被拒绝。把这两个身份拆开，问题就清晰多了。

[配图3：RAM 用户与 RAM 角色职责示意图]

权限问题也是这一轮花时间最多的地方之一。真实系统一旦接入 OSS、STS、函数计算，就不可能靠猜来配置权限。我们遇到过好几类典型错误。比如 STS 能返回临时上传凭证，但后端生成签名读图 URL 时却被 OSS 拒绝；再比如前端拿到临时凭证后可以上传成功，但后端 OCR 阶段读取图片失败；还有一种更隐蔽的情况，是权限已经配了，但配给了错误的主体，比如配到了 `homework` 角色上，而实际读图用的却是 `.env` 里那对 AK/SK 对应的 RAM 用户。最后我们把权限梳理成了最小矩阵：RAM 用户负责 `sts:AssumeRole` 和 `oss:GetObject`，`homework` 角色负责 `oss:PutObject` 和 `oss:AbortMultipartUpload`，角色信任策略再允许 RAM 用户扮演这个角色。权限问题理顺之后，上传、读取、识别三个阶段才真正进入稳定状态。

对象存储这块还有一个非常容易被忽略的细节：Endpoint 和 URL 并不是“能拼出来就能用”。一开始 OSS 地址看上去没问题，甚至对象路径也没问题，但浏览器直传时会报 `XHR error`，服务端读图签名又会报 `SignatureDoesNotMatch` 或 `AccessDenied`。继续往下查才发现，二级域名、三级域名、bucket 域名、区域 endpoint 这几种写法在不同阶段并不能随便混用。尤其是在 V4 签名下，OSS 对 CanonicalRequest 的路径要求非常严格。哪怕 URL 最终长得差不多，只要签名使用的 host、path、query 顺序和 OSS 服务端认定的不一致，就会被拒绝。把这一层问题彻底修完以后，服务端才能稳定生成可读的签名 URL。

[配图4：OSS 上传与签名 URL 调试截图]

再往后，我们遇到的不是云服务配置问题，而是真实模型返回结果和我们代码预期之间的差异。比如 OCR 模型已经调用成功，图片也能识别出来，但返回结果里的 `questionNo` 不是数字，而是字符串。对人来说，`"1"` 和 `1` 没区别；但对严格的 schema 校验来说，这就是两种类型，会直接导致整个批改流程返回 400。这个问题如果只在演示阶段看页面，是根本发现不了的。只有走到了“真实上传 -> 真实 OCR -> 真实解析”这一步，才会暴露出来。最终我们做了兼容：如果模型返回的是 `"1"`、`"9"` 这种数字字符串，就安全转换成数字，再进入后续判分。这类“模型返回格式并不总是完全服从约束”的情况，在真实 AI 集成里非常常见，开发时必须给系统留出合理的兼容层。

还有一个很关键的落地点，是浏览器上传的稳定性。即便服务端和 OSS 都已经配置正确，浏览器侧仍然可能因为 CORS 或预检失败而无法直传。这种情况下，如果产品只能依赖浏览器直传，那老师在页面上看到的仍然会是失败。为了保证体验可验收，我们最终做了一个非常务实的方案：前端优先走 `STS + OSS 直传`，一旦浏览器侧因为 XHR、预检、CORS 等问题失败，就自动回退到后端代理上传，由后端 `/uploads/direct/*` 接收文件，再写入真实 OSS。这样一来，即使浏览器直传没完全调顺，产品功能也不会被卡死。对于真实项目来说，这比死磕“必须先让浏览器直传完美无误”更符合工程常识。

[配图5：浏览器直传失败后自动回退到后端代理上传的流程图]

这一轮开发完成后，首页的验收路径也终于变得简单而明确：输入体验码 `demo-code`，点击“填入演示答案”，上传演示答题卡图片，点击“开始体验”，然后等待系统返回真实批改结果。这个结果不是写死的演示数据，而是真实上传到 OSS 的图片经过真实 OCR 识别、真实判分逻辑处理、再由真实文本模型生成讲评之后的输出。到这一步，我们才可以说，这个产品已经从“会动的 demo”进入了“可继续迭代的真实系统”。

回头看整个过程，最大的收获并不是某一个接口终于返回了 200，而是一个很朴素的工程经验再次被验证：AI 产品的难点从来不只是模型本身，而是模型前后的整条工程链路。文件怎么上传，权限怎么隔离，图片怎么读取，模型返回格式怎么兜底，浏览器异常怎么回退，这些问题如果不解决，哪怕模型再强，用户也用不起来。相反，一旦这些环节都被打通，后面的优化才真正有意义，比如提升 OCR 版式适配范围、增加结果页可视化、引入历史任务管理、扩展老师后台能力等等。

接下来，如果要继续把这个项目往产品级推进，我会优先做三件事。第一，把当前上传、识别、讲评链路的关键日志补全，方便线上排障。第二，把 OSS、RAM、FC、前端部署配置整理成一套标准化文档，降低环境迁移成本。第三，再回到前端体验层，把结果页、讲评卡片、上传提示、异常状态做得更友好。因为真实链路已经通了，后面的每一步优化，都会真正作用在产品本身，而不是继续堆在一个演示壳子上。
