# AI 批改作业独立站 MVP 设计文档

## 1. 项目目标

本项目第一阶段要建设一个移动端优先的 AI 批改作业独立站，用于对外展示产品能力，并提供可实际体验的数学作业批改演示。

站点目标不是一次性做成完整教学平台，而是先上线一个最小可用闭环：

1. 老师打开首页即可理解产品价值。
2. 老师可以低成本录入标准答案并上传固定版式数学答题卡图片。
3. 系统调用真实 AI 完成识别、判分和讲评生成。
4. 页面返回清晰、可信的结果。
5. 页面保留联系方式，承接后续转化。

## 2. 产品定位

### 2.1 核心用户

核心用户是需要快速批改作业的老师，尤其适合使用固定版式题单或答题卡进行练习、随堂测和小测的场景。

### 2.2 第一阶段定位

第一阶段产品同时承担两种角色：

1. 对外展示用的独立站首页
2. 可真实体验的在线批改 demo

因此，首页既要能讲清产品，也要能直接进入体验流程。

### 2.3 当前范围限定

本阶段只支持以下范围：

1. 数学选择题
2. 数学填空题
3. 固定版式答题卡或题单拍照上传

本阶段暂不支持以下范围：

1. 自由版式手写作业本
2. 数学解答过程题
3. 教师后台、班级管理、学生账户体系
4. 历史记录、批量任务、老师复核后台
5. 标准答案图片上传识别

## 3. 设计原则

### 3.1 移动端优先

页面默认按手机访问体验设计。桌面端允许正常浏览，但布局、节奏、交互优先服务手机端。

### 3.2 首页即入口

用户不需要先理解复杂平台结构。首页应直接承接产品介绍、能力说明和演示入口。

### 3.3 明确能力边界

只承诺当前真正能做好的能力，不用模糊表达制造“什么都支持”的错觉。对于不支持或识别不稳定的情况，要明确提示而不是硬判。

### 3.4 先跑通闭环

第一版优先跑通“上传 -> 识别 -> 判分 -> 展示结果 -> 联系转化”，暂不扩展完整作业管理能力。

### 3.5 尽量降低输入成本

老师是否愿意继续使用，取决于首次体验是否足够顺滑。第一版允许老师输入标准答案，但必须把输入成本压到最低，避免把 demo 做成配置工具。

答案录入设计要满足以下原则：

1. 优先让老师复制粘贴一行答案完成配置
2. 提供选择题与填空题分开录入的轻量表单
3. 给出默认示例，减少老师理解成本
4. 支持少量题目快速修改，而不是要求完整后台配置
5. 不让老师先经历复杂建作业流程

## 4. 站点信息架构

本阶段独立站采用轻量结构，核心上控制为 3 个主要视图：

1. 首页
2. 处理中状态
3. 结果页

### 4.1 首页结构

首页从上到下包含以下内容：

1. 首屏
2. 能力说明
3. 演示区
4. 适用边界
5. 联系方式

#### 首屏

首屏承担三件事：

1. 讲清楚价值主张
2. 明确适用题型
3. 引导进入演示或联系咨询

首屏核心表达应为纯中文，避免中英文混排。主按钮采用强动作文案，例如“立即体验批改演示”，次按钮可为“查看适用题型”“添加微信咨询”等。

#### 能力说明

能力说明模块用于快速解释站点当前可完成的真实流程：

1. 上传固定版式答题卡照片
2. AI 识别题号与答案
3. 自动判分并生成讲评

这部分要强调“先做固定版式”是为了提升识别稳定性，体现产品克制和可信度。

#### 演示区

演示区直接嵌入首页下半部分，不要求跳转到复杂功能页。老师进入站点后，应能很快定位上传入口并开始体验。

演示区应包含：

1. 标准答案录入区
2. 上传组件
3. 拍照或选图提示
4. 适用内容说明
5. 结果预期说明

#### 适用边界

该模块用于明确告诉用户当前支持与不支持的内容，避免错误预期。

应重点说明：

1. 当前仅支持固定版式题单和答题卡
2. 当前主要支持数学选择题和填空题
3. 复杂解答题和自由排版作业暂不支持

#### 联系方式

首页底部要保留清晰的联系方式，用于承接真实转化。形式可以包括：

1. 微信
2. 邮箱
3. 预约试用入口

## 5. 页面设计

### 5.1 首页

#### 页面目标

让老师在最短时间内完成两个动作：

1. 理解产品能做什么
2. 开始体验一次真实批改

#### 核心组件

首页需要包含以下核心组件：

1. 价值主张模块
2. 三步流程说明模块
3. 标准答案录入模块
4. 演示上传模块
5. 适用范围模块
6. 联系转化模块

#### 交互要求

1. 上传入口在首屏之后立即可见，不埋太深。
2. 主按钮优先引导体验，而不是引导阅读大量介绍。
3. 页面文案统一使用中文。
4. 交互提示对老师视角友好，例如“建议竖版拍摄”“支持固定版式题单”等。
5. 标准答案录入必须带示例和默认占位，不允许让老师面对空白复杂表单。
6. 先录答案，再传图片，但这两个动作必须合并在同一演示区完成。

#### 标准答案录入设计

首页演示区上半部分需要加入轻量答案录入模块，目标是在 10 到 20 秒内完成配置。

推荐提供两种录入方式：

1. 一行快速录入
2. 分题轻量录入

一行快速录入作为默认方式，适合多数老师第一次使用。示例：

1. 选择题：`1.A 2.C 3.B 4.D`
2. 填空题：`5.12 6.3/4 7.18`

分题轻量录入作为补充方式，用于不习惯文本格式的老师。界面上按题号展示输入框，但只支持小规模题量，不延伸成完整题库管理。

页面需要提供明确引导，例如：

1. “先填标准答案，再上传学生答题卡”
2. “可直接粘贴：1.A 2.C 3.B”
3. “如果是填空题，请直接填写最终答案”

页面还应支持一个默认示例按钮，用来快速填入演示答案，降低首次体验门槛。

### 5.2 处理中状态

#### 页面目标

在上传之后向用户清晰反馈当前系统状态，减少等待焦虑。

#### 处理阶段

处理中的状态拆分为四个阶段：

1. 标准答案已确认
2. 图片上传成功
3. AI 正在识别题目与答案
4. AI 正在生成判分结果和讲评

#### 页面反馈要求

1. 展示明确的进度或状态文案
2. 告知用户预计等待时间
3. 避免让用户误以为页面卡死

### 5.3 结果页

#### 页面目标

结果页必须让用户快速理解这次批改的结论和价值，同时自然承接下一次体验或联系转化。

#### 核心内容

结果页包含以下内容：

1. 总分
2. 批改状态标签
3. 题号正误概览
4. 错题摘要
5. 讲评建议
6. 再次上传按钮
7. 联系正式版入口

#### 展示逻辑

1. 页面顶部先展示总分和整体状态。
2. 中间用题号矩阵展示逐题正误，方便快速扫读。
3. 错题摘要用于解释关键错误。
4. 讲评建议用于呈现老师可直接使用的讲评内容。
5. 页面底部同时给出“再试一张”和“联系正式版”两个动作。

#### 输出示例结构

结果页输出信息可抽象为：

1. 总分：92 分
2. 整体结论：共批改 20 题，建议重点复看 3、6、12 题
3. 题号结果：正确 / 错误
4. 错题说明：标准答案、识别答案、简要建议
5. 讲评建议：按知识点聚合的课堂讲评文案

## 6. 功能范围

### 6.1 本阶段必做功能

1. 移动端优先首页
2. 标准答案快速录入
3. 图片上传
4. 上传预览
5. 调用真实 AI 处理图片
6. 返回结构化判分结果
7. 展示总分、逐题结果、错题摘要、讲评建议
8. 联系方式展示

### 6.2 本阶段不做功能

1. 用户登录
2. 班级管理
3. 作业列表
4. 教师复核流程
5. 学生端查看结果
6. 结果历史归档
7. 多套题配置后台
8. 标准答案图片识别导入

## 7. AI 与数据流设计

### 7.1 处理流程

第一版核心数据流如下：

1. 老师在首页输入或粘贴标准答案
2. 用户在首页上传答题卡图片
3. 前端将答案文本和图片一并传给服务端
4. 服务端调用 OCR 或多模态模型识别题号和答案
5. 服务端将识别结果与老师输入的标准答案比对
6. 服务端生成总分、逐题结果、错题摘要和讲评建议
7. 前端渲染结果页

### 7.2 标准答案策略

第一版支持老师手动输入标准答案，但不做完整后台配置系统，也不做标准答案图片上传识别。

这样做的原因是：

1. 能快速跑通 MVP
2. 能把输入复杂度控制在可接受范围
3. 能把重点放在识别、判分和结果表达上

第一版标准答案录入建议采用以下策略：

1. 默认显示一行快速录入框
2. 提供可直接复制的示例格式
3. 允许老师手动切换到分题录入
4. 支持“填入演示答案”按钮
5. 不要求老师创建作业、配置班级、设置复杂规则

### 7.3 标准答案输入结构

服务端需要先把老师输入的标准答案解析成结构化数据。建议最少支持两类：

1. 选择题：题号 + 单个选项值
2. 填空题：题号 + 文本或数值答案

示例输入：

1. `1.A 2.C 3.B 4.D`
2. `5.12 6.3/4 7.18`

解析后的最小结构可为：

1. 题号
2. 题型
3. 标准答案
4. 判分规则类型

### 7.4 结果数据结构

前端结果展示需要最少拿到以下信息：

1. 本次任务 ID
2. 批改状态
3. 总题数
4. 总分
5. 每题题号
6. 每题标准答案
7. 每题识别答案
8. 每题正误结果
9. 每题简要反馈
10. 整体讲评建议
11. 置信度或稳定性提示

## 8. 异常与边界处理

### 8.1 图片问题

当图片存在以下问题时，不应直接返回高置信度判分：

1. 模糊
2. 裁切不完整
3. 光照过暗
4. 非固定版式

此时应提示用户重新拍摄，并明确原因。

### 8.2 标准答案输入异常

当老师输入的标准答案存在以下问题时，系统应在上传前就拦截：

1. 格式无法解析
2. 题号重复
3. 题号缺失过多
4. 选择题答案不是合法选项

页面应给出尽量直接的提示，例如：

1. “第 3 题答案格式无法识别，请检查是否写成 3.A”
2. “检测到重复题号 6，请确认后再提交”

### 8.3 超范围题型

对于超出当前能力范围的题型，例如复杂解答题或过程题，系统应明确提示“当前版本暂不支持”，而不是输出看似完整但不可靠的结果。

### 8.4 低置信度识别

如果 AI 对部分答案识别不稳定，结果页应给出低置信度提醒。必要时可以提示“建议老师人工复核”。

## 9. 技术方案边界

### 9.1 推荐实现形态

推荐采用：

1. 静态前端站点
2. 轻量服务端 API
3. 第三方对象存储或服务端临时存储
4. 外部 AI 接口

### 9.2 部署建议

前端优先部署在以下任一平台：

1. Cloudflare Pages
2. Vercel

后端使用最小化 API 方案承接：

1. 图片上传
2. AI 调用
3. 结果返回

### 9.3 上线节奏

当前建议的建设顺序如下：

1. 注册 GitHub
2. 购买 `.com` 域名
3. 用 Cloudflare Pages 或 Vercel 部署静态站
4. 先上线首页、联系方式、产品介绍与 demo
5. 再接更多功能页
6. 有用户后，再考虑备案和迁移

## 10. 验收标准

第一阶段上线以以下标准作为验收：

1. 手机端访问体验顺畅
2. 首页价值主张清晰
3. 老师可以低成本完成标准答案录入
4. 用户可以上传真实答题卡图片
5. 系统可以返回真实 AI 批改结果
6. 结果页结构清晰、表达可信
7. 页面保留可联系你的转化入口

## 11. 后续扩展方向

当第一版跑通后，可逐步扩展：

1. 支持更多数学题型
2. 支持更多固定模板
3. 增加教师登录和作业记录
4. 增加老师复核流程
5. 增加产品介绍和功能详情页
6. 再建设完整个人站和品牌站
