# 班级单题批量批改设计文档

## 1. 目标

在当前固定版式客观题批改能力之外，新增一条独立能力线：支持老师上传“同一道题的班级答案 PDF”，再结合评分标准图片或 PDF，对自由排版、手写过程题进行逐页评分，并输出班级汇总表与逐页批注结果。

这一能力不是替代现有 MVP，而是新增第二个产品页面。现有固定版式批改维持现状，不在本轮重构。

## 2. 产品定位

### 2.1 页面定位

新能力采用独立页面，不与现有固定版式批改页混排。原因如下：

1. 输入材料完全不同
2. 任务时长完全不同
3. 输出结构完全不同
4. 移动端认知负担会显著上升

因此产品结构调整为：

1. 固定版式快速批改页
2. 班级单题批量批改页

首页后续只需要提供两个清晰入口，不在同一演示区混合两套流程。

### 2.2 目标用户

核心用户仍然是老师，但使用场景从“快速批改客观题”扩展到“批量评阅同一道开放题/过程题”：

1. 一个班围绕同一道题完成作答
2. 老师已有整班 PDF 汇总材料
3. 评分依据已经以 rubric、参考答案、分层样例图等形式存在

### 2.3 第一版范围

第一版只支持以下情形：

1. 一个 PDF 对应同一道题的全班作答
2. PDF 每页是一份学生作答
3. 题目内容出现在学生作答页内
4. 老师额外上传评分标准图片或 PDF
5. 输出逐页评分与班级汇总

第一版暂不支持：

1. 一个 PDF 中混合多道题
2. 自动识别真实学生姓名并建立名单映射
3. 老师在线手工复核改分后台
4. 多题任务合并统计
5. 最终成绩单导出模板定制
6. 通用作业本整册自由识别

## 3. 输入材料设计

### 3.1 班级答案 PDF

系统将班级答案 PDF 视为批量任务主材料。

要求如下：

1. 同一道题的班级答案合集
2. 默认一页对应一份学生作答
3. 页面内包含题目内容或题干截图
4. 页面可能是自由排版手写内容
5. 页面可能包含图示、线段图、计算过程、文字解释

第一版不尝试把 PDF 解析为结构化文本，而是先转为逐页图像，再做多模态理解。

### 3.2 评分标准材料

老师必须上传评分标准图片或 PDF，系统不走纯自动裸判模式。

评分标准材料可能包括：

1. 参考答案
2. 达到预期样例
3. 超出预期样例
4. 关键得分点
5. 常见错误提示
6. 分层评价标准

系统应把评分标准材料作为与学生答案同等重要的 AI 输入，而不是仅作附加备注。

### 3.3 题目来源

题目默认从学生作答页中获取，不要求老师单独再传题目图片。

这意味着逐页理解时，模型需要同时看：

1. 当前页题目
2. 当前页学生作答
3. 全局评分标准材料

## 4. 页面与流程设计

### 4.1 独立页面

新能力单独使用一个移动端优先页面，例如：

1. `/batch-review`

该页面风格沿用现有站点视觉语言，但不复用原有首页演示布局。

### 4.2 交互形式

采用向导式分步页，而不是单页堆叠表单。原因如下：

1. PDF 与 rubric 输入复杂度高
2. 移动端单屏信息密度必须降低
3. 后续可自然加入预览、页数确认、处理中状态

### 4.3 分步结构

第一版建议拆成 3 步：

#### 第 1 步：上传班级答案 PDF

页面目标：

1. 让老师只做一件事：上传班级答案 PDF
2. 明确该 PDF 必须是“同一道题的整班作答合集”
3. 允许展示页数、文件名和大小

#### 第 2 步：上传评分标准材料

页面目标：

1. 上传 rubric 图片或 PDF
2. 明确这是 AI 评分依据
3. 提示老师可上传“达到预期 / 超出预期 / 参考解法”类材料

#### 第 3 步：确认并开始批改

页面目标：

1. 展示任务摘要
2. 显示 PDF 页数
3. 显示评分标准材料已就绪
4. 让老师明确知道系统将按页逐个学生评分

按钮文案应类似：

1. 开始批量批改

### 4.4 处理中状态

批量任务需要单独的处理中页或状态卡片，不能复用现有单次批改 4 步提示。

建议阶段：

1. 任务已创建
2. 正在拆分 PDF 页面
3. 正在解析评分标准
4. 正在逐页理解学生作答
5. 正在生成班级汇总

### 4.5 结果页

结果页分为两层：

1. 班级汇总
2. 逐页学生结果

移动端默认先看班级汇总，再展开逐页详情。

## 5. 输出结果设计

### 5.1 班级汇总表

班级汇总表第一版先按页序展示，不强依赖姓名识别。

记录字段建议包括：

1. `第 N 份`
2. 分数
3. 等级
4. 一句话问题总结

示例：

1. 第 1 份 / 8 分 / 达到预期 / 图示完整，但数量关系说明不够清楚
2. 第 2 份 / 5 分 / 待提升 / 找到了部分关系，但两种情况未说明完整

### 5.2 逐页学生结果

逐页详情风格采用“老师批注型”，而不是纯模型分析报告。

每页至少输出：

1. 分数
2. 等级
3. 做得好的地方
4. 主要问题
5. 改进建议

可选附加：

1. 是否达到预期
2. 是否覆盖两种情况
3. 是否有图示表达

### 5.3 等级体系

第一版建议使用低歧义等级，而不是过多花哨标签。

例如：

1. 超出预期
2. 达到预期
3. 基本达到
4. 待提升

等级来源应以老师上传 rubric 为主，模型只负责映射和解释，不自行发明标准。

## 6. AI 处理链路

### 6.1 核心原则

这个能力不再是“标准答案比对”，而是“题目 + rubric + 学生过程”三方联合理解。

### 6.2 建议链路

第一版采用单任务、多页批处理模式：

1. 上传班级答案 PDF 到对象存储
2. 上传评分标准材料到对象存储
3. 服务端将 PDF 拆页为多张图
4. 服务端提取或归档评分标准图片页
5. 逐页调用多模态模型，输入：
   - 当前页学生作答
   - 当前页题目
   - 评分标准材料
6. 得到结构化评分结果
7. 汇总为班级总表

### 6.3 模型职责拆分

建议保留双模型或分阶段处理：

1. 多模态模型负责看图理解、过程判断、结构化评分
2. 文本模型负责把结构化结论整理成更稳定的老师批注文案

### 6.4 结构化输出

逐页模型输出不应只是一段自由文本，而应包含结构字段，例如：

1. `score`
2. `level`
3. `strengths[]`
4. `issues[]`
5. `suggestions[]`
6. `summary`

这样才能稳定生成班级汇总表，并支持后续复核能力。

## 7. 后端任务模型

### 7.1 新任务类型

当前系统已有单次批改链路。新功能需要新增任务类型，例如：

1. `single_sheet_review`
2. `batch_freeform_review`

### 7.2 批量任务实体

批量任务至少要保存：

1. 任务 ID
2. 上传的班级答案 PDF 对象键
3. 上传的评分标准材料对象键
4. PDF 页数
5. 当前处理状态
6. 逐页结果数组
7. 汇总结果

### 7.3 页级结果实体

每页结果建议独立建模，便于分页与复核：

1. `pageNo`
2. `displayName`，第一版固定为“第 N 份”
3. `score`
4. `level`
5. `summary`
6. `strengths`
7. `issues`
8. `suggestions`

## 8. 前端信息架构

建议新增如下页面：

1. 批量批改任务创建页
2. 批量任务处理中页
3. 批量任务结果页

首页后续只提供两个清晰入口：

1. 固定版式快速批改
2. 班级批量批改

不在同一操作区混合两类流程。

## 9. 验收标准

第一版验收通过的定义：

1. 老师可以上传真实班级答案 PDF
2. 老师可以上传真实评分标准图片或 PDF
3. 系统能按页处理全部学生作答
4. 每页都能返回结构化评分结果
5. 结果页能看到班级汇总表
6. 结果页能查看逐页“老师批注型”详情

## 10. 明确不做的事

为了保证第一版可落地，本轮明确不做：

1. 与现有固定版式批改页合并
2. 自动识别学生真实姓名
3. 多道题批量任务
4. 老师在线改分与复核工作台
5. 通用整本作业自由识别
6. 班级维度长期档案管理

## 11. 风险与应对

### 11.1 评分稳定性风险

自由排版过程题天然比客观题更不稳定。应对方式：

1. 强制上传评分标准材料
2. 把“逐页结构化结果”作为中间层
3. 保留等级和问题标签，避免只有长文本

### 11.2 PDF 质量风险

班级 PDF 可能来源复杂。应对方式：

1. 第一版只支持图像型 PDF 逐页拆分
2. 上传前做页数、大小和格式校验
3. 对异常页返回“识别不充分”状态

### 11.3 移动端可用性风险

批量任务天然更复杂。应对方式：

1. 采用向导式分步页
2. 一屏只做一个动作
3. 结果页先看汇总，再看逐页详情
